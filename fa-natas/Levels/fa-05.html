۰۵ دستکاری کوکی — وضعیت احراز هویت سمت کاربر متوسط
// مفاهیم کلیدی
دستکاری کوکی اعتماد به سمت کاربر امنیت نشست دور زدن احراز هویت ابزارهای توسعه‌دهنده
// مرور کلی چالش

صفحه پیام "Access disallowed. You are not logged in." را نمایش می‌دهد. بررسی کوکی‌های مرورگر نشان می‌دهد که سرور وضعیت ورود را با استفاده از یک مقدار کوکی ساده که کاربر می‌تواند آزادانه آن را تغییر دهد، پیگیری می‌کند.

Cookie: loggedin=0
// تحلیل اولیه

کوکی‌ها قطعات کوچکی از داده هستند که در مرورگر ذخیره می‌شوند و با هر درخواست HTTP به سرور ارسال می‌شوند. معمولاً برای موارد زیر استفاده می‌شوند:

  • مدیریت نشست
  • تنظیمات کاربر
  • ردیابی و تحلیل

در این چالش، سرور از یک کوکی به نام loggedin برای تعیین احراز هویت کاربر استفاده می‌کند. مقدار 0 به معنای "وارد نشده" و مقدار 1 به معنای "وارد شده" است. این یک آسیب‌پذیری کلاسیک دور زدن احراز هویت است زیرا کوکی به صورت رمزنگاری‌شده محافظت نمی‌شود.

💡 کوکی‌های سمت کاربر یک مکانیزم قابل اعتماد برای کنترل دسترسی نیستند. منطق احراز هویت باید در سمت سرور اعمال شود. هر داده‌ای که کاربر می‌تواند بخواند یا بنویسد هرگز نباید برای تصمیمات امنیتی مورد اعتماد قرار گیرد.
// مراحل حل

روش‌های متعددی برای تغییر مقدار کوکی وجود دارد:

روش ۱ — استفاده از ابزارهای توسعه‌دهنده مرورگر (توصیه‌شده):

  1. DevTools را با F12 باز کنید.
  2. به برگه Application (کروم/اج) یا Storage (فایرفاکس) بروید.
  3. در بخش Cookies، http://natas5.natas.labs.overthewire.org را انتخاب کنید.
  4. کوکی loggedin را پیدا کرده و مقدار آن را از 0 به 1 تغییر دهید.
  5. صفحه را بازخوانی کنید.

روش ۲ — استفاده از کنسول جاوااسکریپت:

  1. کنسول را در DevTools باز کنید.
  2. جاوااسکریپت زیر را اجرا کنید:
document.cookie = "loggedin=1"; location.reload();

روش ۳ — استفاده از curl:

curl -u natas5:iX6IOfmpN7AYOQGPwtn3fXpbaJVJcHfq \ --cookie "loggedin=1" \ http://natas5.natas.labs.overthewire.org/

روش ۴ — استفاده از افزونه‌های مرورگر:

افزونه‌هایی مانند EditThisCookie یا Cookie-Editor یک رابط کاربری ساده برای تغییر کوکی‌ها فراهم می‌کنند.

روش ۵ — استفاده از Python (کتابخانه requests):

import requests url = "http://natas5.natas.labs.overthewire.org/" auth = ("natas5", "iX6IOfmpN7AYOQGPwtn3fXpbaJVJcHfq") cookies = {"loggedin": "1"} response = requests.get(url, auth=auth, cookies=cookies) print(response.text)

پس از تغییر کوکی، صفحه رمز عبور سطح بعدی را نمایش خواهد داد.

# پس از تغییر کوکی، خواهید دید: Access granted. The password for natas6 is aGoY4q2Dc6MgDq4oL4YtoKtyAg9PeHa1
// چرا کار می‌کند

سرور مقدار کوکی loggedin را بررسی می‌کند و در صورت برابر بودن با 1 دسترسی می‌دهد. با این حال، چندین مشکل اساسی در این رویکرد وجود دارد:

  • بدون یکپارچگی رمزنگاری: کوکی امضا یا رمزگذاری نشده است.
  • مقدار قابل پیش‌بینی: 0 و 1 به راحتی قابل حدس هستند.
  • تحت کنترل کلاینت: کاربران می‌توانند با استفاده از ابزارهای داخلی مرورگر کوکی‌ها را تغییر دهند.
  • بدون اعتبارسنجی نشست: سرور کوکی را در برابر یک فروشگاه نشست سمت سرور اعتبارسنجی نمی‌کند.
// مشکل وضعیت سمت کاربر

وقتی وضعیت احراز هویت در سمت کاربر بدون محافظت رمزنگاری ذخیره می‌شود، مهاجمان می‌توانند:

  • مقادیر کوکی را برای جعل هویت کاربران دیگر تغییر دهند
  • با تغییر پرچم‌های نقش یا مجوز، سطح دسترسی را افزایش دهند
  • احراز هویت را کاملاً دور بزنند
  • حملات تثبیت نشست (Session Fixation) انجام دهند
// اقدامات دفاعی
  • هرگز به داده‌های سمت کاربر برای تصمیمات احراز هویت اعتماد نکنید.
  • از نشست‌های سمت سرور با شناسه‌های نشست تصادفی و رمزنگاری‌شده استفاده کنید.
  • نشست‌ها را در یک فروشگاه سمت سرور ذخیره کنید (مانند پایگاه داده، Redis، حافظه).
  • از پرچم‌های HttpOnly و Secure روی کوکی‌ها استفاده کنید.
  • مکانیزم‌های انقضای نشست و باطل‌سازی مناسب پیاده‌سازی کنید.
// مشخصات ورود
natas5 iX6IOfmpN7AYOQGPwtn3fXpbaJVJcHfq
natas6 aGoY4q2Dc6MgDq4oL4YtoKtyAg9PeHa1
// نکات کلیدی
🍪 کوکی‌ها تحت کنترل کاربر هستند

کاربران می‌توانند با استفاده از ابزارهای مرورگر کوکی‌ها را تغییر دهند. هرگز بدون محافظت رمزنگاری از آنها برای احراز هویت استفاده نکنید.

🔐 نشست‌های سمت سرور

وضعیت احراز هویت همیشه باید در سمت سرور اعتبارسنجی شود. از شناسه‌های نشست رمزنگاری‌شده استفاده کنید.
