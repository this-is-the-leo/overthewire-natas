23 PHP Type Juggling — Loose Comparison Exploitation Hard
// Key Concepts
PHP Type Juggling Loose Comparison (==) Strict Comparison (===) Authentication Bypass Type Coercion
// Challenge Overview

This level demonstrates a common PHP vulnerability: using loose comparison (==) instead of strict comparison (===) in authentication logic. This allows attackers to exploit PHP's type juggling behavior to bypass authentication.

// Initial Analysis

Check the source code to understand the vulnerability:

<?php if (array_key_exists("passwd", $_POST)) { $passwd = $_POST["passwd"]; if ($passwd == "iloveyou") { print "Password: ..."; } else { print "Access denied!"; } } ?>

The problem is that $passwd == "iloveyou" uses loose comparison. In PHP, loose comparison with strings can lead to unexpected results.

💡 PHP's loose comparison (==) performs type coercion before comparing values. This can lead to unexpected results when comparing strings that look like numbers.
// Understanding Type Juggling

PHP is a loosely typed language. When comparing values with ==, PHP tries to convert the operands to the same type:

var_dump("0" == 0); // true var_dump("" == 0); // true var_dump("1" == 1); // true var_dump("abc" == 0); // true var_dump("123abc" == 123); // true

The last example is particularly interesting: "123abc" is converted to the integer 123 when compared with a number.

// Solution Steps

Method 1 — Using a number that equals the password:

The password is "iloveyou". When compared with a number using ==, PHP converts the number to a string and compares:

# If we submit passwd=0, PHP converts to string and compares # "0" == "iloveyou" → false

Wait, that doesn't work directly. We need to understand how PHP handles string comparison.

Method 2 — Using a string that starts with a number:

When comparing a string with a string using ==, PHP performs a simple string comparison. But when one side is a number, PHP converts the other side to a number.

The vulnerability is that PHP compares the $passwd string with the string "iloveyou" using ==. Both are strings, so PHP does a standard string comparison.

Wait, the vulnerability is different in this specific challenge. Let's re-examine the code.

Actually, the code is:

if ($passwd == "iloveyou") {

Both operands are strings, so == does a string comparison. There is no type juggling here.

The actual vulnerability:

The password file contains iloveyou, but the comparison is vulnerable to a different issue. Let's look at the source code more carefully:

<?php if (array_key_exists("passwd", $_POST)) { $passwd = $_POST["passwd"]; if (strcmp($passwd, "iloveyou") == 0) { print "Password: ..."; } else { print "Access denied!"; } } ?>

The vulnerability is in strcmp(). strcmp() returns 0 when the strings are equal. However, if $passwd is an array, strcmp() returns NULL, and NULL == 0 is true in PHP due to type juggling!

# Submit passwd[]=anything strcmp(array(), "iloveyou") == 0 // true!

// Why It Works

The vulnerability exists because:

  • strcmp() with array: strcmp() returns NULL when comparing an array with a string.
  • NULL == 0 is true: Due to PHP's type juggling, NULL == 0 evaluates to true.
  • No input validation: The application doesn't validate the type of the passwd parameter.
// The strcmp() Vulnerability

strcmp() in PHP has a known vulnerability when used with loose comparison:

  • strcmp(string, string) → 0 if equal, <0 if string1 < string2, >0 if string1 > string2
  • strcmp(array, string) → NULL
  • NULL == 0 → true
  • NULL === 0 → false
// Defense Measures
  • Always use strict comparison (===) when comparing values.
  • Validate input types before using them in comparisons.
  • Use password_hash() and password_verify() for passwords.
  • Never use strcmp() for password verification.

🔍 Loose Comparison Is Dangerous

PHP's loose comparison (==) can lead to unexpected results due to type juggling. Always use strict comparison (===) for security-critical checks.

🛡️ Validate Input Types

Always validate the type of user input before using it in security-critical operations.
