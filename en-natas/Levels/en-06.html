06 PHP Include — Code Injection Hard
// Key Concepts
PHP Include Local File Inclusion (LFI) Code Injection Input Validation Parameter Manipulation
// Challenge Overview

This level presents a form with an input field for a "secret" value. There is also a hidden clue in the page — inspecting the source reveals a reference to a PHP file that is being included. The vulnerability is in the include function, which dynamically loads files based on user input.

// Initial Analysis

To understand how the application works, check the source code:

<form method="post"> Input secret: <input name="secret" type="text"> <input type="submit" value="Submit"> </form> <!-- include includes/secret.inc -->

The comment reveals that the program includes a PHP file: includes/secret.inc. This suggests that the secret parameter might be used to dynamically include files.

💡 In PHP, the include() function executes the contents of a specified file. If user input is used to determine the file, an attacker can load arbitrary files — including source code or sensitive configuration files.
// Understanding PHP Include

The include and require functions in PHP are used to insert the contents of one file into another. They are commonly used for:

  • Reusing common code (headers, footers, navigation)
  • Loading configuration files
  • Organizing code into modules

However, when the file name is taken from user input without proper validation, it becomes a Local File Inclusion (LFI) vulnerability.

// Solution Steps
  1. Inspect the page source and find the include comment:
<!-- include includes/secret.inc -->
  1. Try to open the included file directly:
http://natas6.natas.labs.overthewire.org/includes/secret.inc
  1. PHP files are typically executed, not displayed. However, you can use the view-source: trick to see the code:
view-source:http://natas6.natas.labs.overthewire.org/includes/secret.inc
  1. You'll see the PHP source code containing the secret:
<?php $secret = "FOEIUWGHFEEUHOFUOIU"; ?>
  1. Enter the secret value in the form on the main page.
  2. The page will display the password for the next level.

Alternative Method — Using curl:

curl -u natas6:aGoY4q2Dc6MgDq4oL4YtoKtyAg9PeHa1 \ --data "secret=FOEIUWGHFEEUHOFUOIU" \ http://natas6.natas.labs.overthewire.org/
// Why It Works

The vulnerability exists for several reasons:

  • User input controls the file path: The secret parameter is used to build the file path.
  • No input validation: The application does not sanitize or validate user input.
  • Exposed file directory: The includes/ directory is publicly accessible.
  • Source code disclosure: Using view-source: reveals PHP code.
// LFI Risks

Local File Inclusion can lead to:

  • Source code disclosure: Reading sensitive PHP files
  • Configuration exposure: Reading files like /etc/passwd
  • Remote Code Execution (RCE): When combined with file upload
  • Privilege escalation: Accessing restricted functionality
// Defense Measures
  • Never use user input to construct file paths.
  • Use a whitelist of allowed file names instead of dynamic inclusion.
  • Store sensitive files (like .inc files) outside the web root.
  • Disable directory listing to prevent file discovery.
  • Use proper error handling to prevent information leakage.
  • Implement input validation for all user-supplied data.

// Key Takeaways
📁 Never Trust User Input for File Paths

Using user input to determine which files to include is extremely dangerous. Always use a whitelist approach.

🔒 Keep Sensitive Files Outside Web Root

Files containing secrets, configuration, or source code should be stored outside the public directory.
