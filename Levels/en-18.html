18 Session ID Prediction — Brute Force Hard

// Key Concepts

Session ID Prediction Brute Force Weak Session Management PHP Session Authentication Bypass

// Challenge Overview

This level uses a session ID to determine if the user is logged in as an administrator. However, the session IDs are generated sequentially and are predictable. By brute-forcing the session ID, we can gain administrative access.

// Initial Analysis

View the source code:

<?php $maxid = 640; if (isset($_COOKIE["PHPSESSID"])) { $sessionid = $_COOKIE["PHPSESSID"]; } else { session_start(); $sessionid = session_id(); setcookie("PHPSESSID", $sessionid); } if ($sessionid >= 1 && $sessionid <= $maxid) { $user = "SELECT * FROM users WHERE id = " . $sessionid; $result = mysql_query($user); if (mysql_num_rows($result) > 0) { $row = mysql_fetch_assoc($result); if ($row['username'] == 'admin') { print "Password: ..."; } } } ?>

The application uses the session ID as a numeric ID to query the database. The admin user likely has a specific ID.

💡 Session IDs should be cryptographically random and unpredictable. Using sequential numeric IDs makes the application vulnerable to brute force attacks.

// Solution Walkthrough

Step 1 — Identify the admin ID:

We need to test each possible session ID (1 to 640) to find the admin user.

Step 3 — Manually test a few IDs:

Step 4 — Use Burp Suite Intruder:

  3. Set the PHPSESSID cookie value as the payload position.

// Why It Works

The session management is vulnerable because:

  • Sequential IDs: Session IDs are numeric and predictable.
  • Small range: Only 640 possible values, easily brute-forced.
  • No authentication: The application doesn't validate the session ID properly.
  • Direct database query: The session ID is used directly in a SQL query.

// Defensive Measures

  • Use cryptographically random session IDs (e.g., PHP's built-in session management).
  • Use a large ID space to prevent brute force (e.g., 128-bit random).
  • Validate sessions server-side — store session data in a database or memory.
  • Implement rate limiting to prevent brute force attempts.
  • Use secure session flags (HttpOnly, Secure, SameSite).

// Secure Session Management Example

<?php // Use PHP's built-in session management session_start(); // Generate a random session ID session_regenerate_id(true); // Store session data on the server $_SESSION['user_id'] = $user_id; // Validate the session on each request if (isset($_SESSION['user_id'])) { $user_id = $_SESSION['user_id']; // Query the database with the validated user ID } ?>

// Key Takeaways

🔑 Session IDs Must Be Random

Sequential or predictable session IDs can be brute-forced. Always use cryptographically random session IDs.

🛡️ Server-Side Session Storage

Store session data on the server, not just in the client cookie. This prevents tampering and improves security.
