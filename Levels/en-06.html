// PHP Include — Code Injection via URL Parameter

🚀 Crafted by Leo ⚡ GitHub · LinkedIn · Telegram
06 PHP Include — Code Injection Medium

// Key Concepts

PHP Include Local File Inclusion (LFI) Code Injection Input Validation Parameter Tampering

// Challenge Overview

This level presents a form with an input field for a "secret" value. The page also has a hidden hint — viewing the source reveals a reference to a PHP file being included. The vulnerability lies in the include function that dynamically loads files based on user input.

// Initial Analysis

View the page source to understand how the application works:

<form method="post"> Input secret: <input name="secret" type="text"> <input type="submit" value="Submit"> </form> <!-- include includes/secret.inc -->

The comment reveals that the application includes a PHP file: includes/secret.inc. This suggests that the secret parameter might be used to dynamically include files.

💡 In PHP, the include() function executes the contents of a specified file. If user input is used to determine which file to include, an attacker can load arbitrary files — including source code or sensitive configuration files.

// Understanding PHP Include

PHP's include and require functions are used to insert the contents of one file into another. They are commonly used for:

  • Reusing common code (headers, footers, navigation)
  • Loading configuration files
  • Organizing code into modules

However, when the filename is taken from user input without proper validation, it becomes a Local File Inclusion (LFI) vulnerability.

// Solution Walkthrough

  1. Inspect the page source and note the include comment:
<!-- include includes/secret.inc -->
  1. Try to access the included file directly:
http://natas6.natas.labs.overthewire.org/includes/secret.inc
  1. PHP files are normally executed, not displayed. But you can use the view-source: trick to see the code:
view-source:http://natas6.natas.labs.overthewire.org/includes/secret.inc
  1. You'll see the PHP source code containing the secret:
<?php $secret = "FOEIUWGHFEEUHOFUOIU"; ?>
  1. Enter the secret value into the form on the main page.
  2. The page will display the password for the next level.

Alternative Method — Using curl:

curl -u natas6:aGoY4q2Dc6MgDq4oL4YtoKtyAg9PeHa1 \ --data "secret=FOEIUWGHFEEUHOFUOIU" \ http://natas6.natas.labs.overthewire.org/

// Why It Works

The vulnerability exists because:

  • User input controls file inclusion: The secret parameter is used to construct a file path.
  • No input validation: The application does not sanitize or validate the user input.
  • File exposure: The includes/ directory is publicly accessible.
  • Source code disclosure: Using view-source: reveals the PHP code.

// The Danger of LFI

Local File Inclusion can lead to:

  • Source code disclosure: Reading sensitive PHP files
  • Configuration exposure: Reading files like /etc/passwd
  • Remote Code Execution (RCE): If combined with file uploads or log poisoning
  • Privilege escalation: Accessing restricted functionality

// Defensive Measures

  • Never use user input to construct file paths.
  • Use a whitelist of allowed filenames instead of dynamic inclusion.
  • Store sensitive files (like .inc files) outside the web root.
  • Disable directory listing to prevent file enumeration.
  • Use proper error handling to prevent information leakage.
  • Implement input validation and sanitization for all user-supplied data.
  • Use allow_url_include = Off in php.ini to prevent remote file inclusion.

// Security Best Practice: File Storage

✅ Correct approach:

  • Store sensitive files outside the document root (e.g., /var/www/includes/ vs /var/www/html/includes/).
  • Use PHP's include path to reference files without exposing their location.
  • Define secret values in environment variables or configuration files that are not web-accessible.

// Key Takeaways

📁 Never Trust User Input for File Paths

Using user input to determine which files to include is extremely dangerous. Always use a whitelist approach.

🔒 Keep Sensitive Files Out of Web Root

Files containing secrets, configuration, or source code should be stored outside the publicly accessible directory.
