// Challenge Overview

This level uses a cookie to store user data that is encrypted with XOR encryption. The cookie contains the user's preferred background color and a flag indicating whether the user is logged in. By understanding the XOR encryption algorithm and extracting the key, we can modify the cookie to gain access.

// Initial Analysis

View the source code to understand the encryption:

<?php $defaultdata = array( "showpassword" => "no", "bgcolor" => "#ffffff"); function xor_encrypt($in) { $key = '??????????????????????????????????'; $text = $in; $outText = ''; for($i=0; $i < strlen($text); $i++) { $outText .= $text[$i] ^ $key[$i % strlen($key)]; } return $outText; } function loadData($d) { $arr = json_decode(xor_encrypt(base64_decode($d)), true); if($arr) return $arr; die("invalid data"); } if(array_key_exists("data", $_COOKIE)) { $data = loadData($_COOKIE["data"]); } else { $data = $defaultdata; } if($data["showpassword"] == "yes") { print "Password: ..."; } ?>

The cookie contains Base64-encoded data that is XOR-encrypted with a secret key. The goal is to create a new cookie where showpassword is set to "yes".

💡 XOR encryption with a repeating key is weak. If we know a plaintext-ciphertext pair, we can recover the key and use it to encrypt any data we want.

// Understanding XOR Encryption

XOR is a reversible operation:

  • Plaintext ^ Key = Ciphertext
  • Ciphertext ^ Key = Plaintext
  • Plaintext ^ Ciphertext = Key (if we know both!)

Since we know the default data (the plaintext) and we have the cookie (the ciphertext), we can recover the key.

// Why It Works

XOR encryption with a repeating key is vulnerable because:

  • Known plaintext attack: We know the default data structure.
  • Repeating key: The key is reused, allowing us to recover it.
  • No integrity protection: The encrypted data can be modified without detection.

// Defensive Measures

  • Use strong encryption like AES-256-GCM with proper key management.
  • Never roll your own crypto — use well-tested libraries.
  • Use a unique key and never reuse it across different data.
  • Include a Message Authentication Code (MAC) to ensure integrity.
  • Store sensitive data on the server instead of in client-side cookies.

// Key Takeaways

🔐 Don't Roll Your Own Crypto

XOR encryption with a repeating key is weak and easily broken. Use standard, well-tested cryptographic libraries.

📝 Client-Side Data Must Be Verified

Any data stored in cookies or local storage must be cryptographically protected and verified on the server.
