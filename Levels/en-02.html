// Key Concepts

// Challenge Overview

The page reads "There is nothing on this page." However, inspecting the HTML source reveals a 1×1 pixel image loaded from /files/pixel.png, exposing the existence of a files directory on the server.

// Initial Analysis

When a web server receives a request for a directory, the default behavior depends on its configuration. If directory listing (also called directory indexing) is enabled, the server returns a list of all files and subdirectories in that folder. This is often enabled by default on misconfigured servers and can expose sensitive files.

💡 A single image tag with a relative path revealed the existence of a /files/ directory. This is called information disclosure — small clues that lead to larger discoveries.

// Solution Walkthrough

  1. View the page source and identify the image tag:
<img src="/files/pixel.png">
  1. Navigate to the parent directory in your browser:
http://natas2.natas.labs.overthewire.org/files/
  1. Directory listing is enabled — you'll see a list of files in the directory.
  2. users.txt is visible in the file list — open it to retrieve the password.

http://natas2.natas.labs.overthewire.org/files/users.txt
# Username:Password natas3:sJIJNW6ucpu6HPZ1ZAchaDtwd7oGrD14

// Why It Works

Web servers like Apache and Nginx can be configured to show a directory index when no default file (index.html, index.php, etc.) is present. When this feature is enabled:

  • All files in the directory become publicly accessible
  • File names often reveal sensitive information
  • Attackers can use this to map the server's file structure

// Defensive Measures

  • Disable directory listing in your web server configuration (e.g., Options -Indexes in Apache).
  • Place an index.html file in every directory to prevent listing.
  • Store sensitive files outside the web root when possible.
  • Use proper access controls and authentication for all resources.
